应用启动详解

注意： 本文基于 Android 9(P) 分析，涉及大量源码，可能引起不适。

一、前言

本文旨在详细展示冷启动过程中，AMS、PMS、WMS的系统工作交互。

二、Launcher 请求 AMS 过程

Android 系统启动后桌面程序就是 Lanucher, 通过 Lanuncher 可以点击图标启动任何一个 App，继而启动它的 Main Activity; 整个过程发生在 Luancher#startActivitySafely 中，最终都回调用到 startActivityForResult，而启动一个 Activity 必须通过系统服务来执行，因此会使用 Launcher 的 ApplicationThread 来发起一次 IPC 调用，最终在 Instrumentation#execStartActivity 中请求 AMS 启动 Activity。

1、起点：HLauncher 发起启动请求

当用户点击应用图标时，HLauncher 作为系统桌面应用，通过 startActivity() 启动目标应用：

startActivitySafely用于安全启动 Activity, 处理了多种情况，包括安全模式检查、用户权限验证、快捷方式处理等；

优化提示：用户点击应用图标时，onClick 响应触发 startActivity() ，正常 onClick 事件是 ACTION_DOWN -> ACTION_UP -> onClick ; 从down到up触发一般有60-90m左右间隔事件；所以可以考虑从down事件开始假启动；

2、向系统服务发送起请求

HLauncher 通过 Binder IPC 调用 ActivityManagerService(AMS)，从 Android Q(10)开始则使用 ActivityTaskManagerService(ATMS);

Activity.java#startActivity() 中通过 Instrumentation 桥接系统调用：在 execStartActivity 中使用 ActivityManager.getService()(通过 Context.ACTIVITY_SERVICE 向 ServiceManager 查询) 获取 AMS ;

三、AMS-Zygote 调用

Launcher 请求 AMS 后，AMS 会判断权限信息，只有具有权限的调用进程才可以启动 Activity；接着 ActivityStarter 解析intent的内容，创建对应的 ActivityRecord 对象，根据是否需要新建 Task 来新建或者选择对应的 TaskRecord 添加 Activity, 并处理 Task 相关状态信息，例如前后台的转移；然后就会使前台Task的栈顶Activity可见，这个过程就会启动对应的Activity，必要情况会启动对应的进程。最后通过IPC调用将启动Activity的事务发送到对应进程中，交由对应App来处理。

1、AMS.startActivity

frameworks\base\services\core\java\com\android\server\am\ActivityManagerService.java#startActivity()

frameworks\base\services\core\java\com\android\server\am\ActivityManagerService.java#startActivityAsUser()

ActivityStarter是Android 7.0中新加入的类，它是加载Activity的控制类，会收集所有的逻辑来决定如何将 Intent和Flags转换为Activity，并将 Activity和Task以及Stack相关联。

frameworks\base\services\core\java\com\android\server\am\ActivityStarter.java#execute()

frameworks\base\services\core\java\com\android\server\am\ActivityStarter.java#startActivityMayWait()

startActivity 将要启动的 Activity 创建 ActivityRecord 对象。ActivityRecord 构建的同时会根据 ActivityRecord 和 intent 创建一个 appToken。appToken是一个IBinder对象，用于提供给WMS关联Window和Activity，后面文章会详细分析。startActivity调用了另一个startActivity的重载，继而调用到了startActivityUnchecked 方法。

startActivityUnchecked主要处理任务栈相关，处理singleTop、singleInstance问题，计算Intent Flag，以及栈顶复用问题。接着调用了调用了ActivityStackSupervisor 的resumeFocusedStackTopActivityLocked 方法。

frameworks\base\services\core\java\com\android\server\am\ActivityStackSupervisor.java#resumeFocusedStackTopActivityLocked()

接下来由 Rframeworks\base\services\core\java\com\android\server\am\ActivityStack.java#resumeTopActivityUncheckedLocked 调用 resumeTopActivityInnerLocked。

resumeTopActivityInnerLocked 方法的逻辑非常复杂，主要是判断当前是否真的需要resume栈顶Activity。简洁说明冷启动情况：当我们的栈顶activity没有绑定到任何Application的时候，就会调用ActivityStackSupervisor#startSpecificActivityLocked。

2、Process

frameworks\base\core\java\android\os\Process.java

接着在 frameworks\base\core\java\android\os\ZygoteProcess.java 中处理：

start函数中只调用了startViaZygote方法：

在注释1处创建了字符串列表argsForZygote ，并将启动应用进程的启动参数保存在argsForZygote中，函数的最后会调用zygoteSendArgsAndGetResult函数,需要注意的是，zygoteSendArgsAndGetResult函数中第一个参数中调用了openZygoteSocketIfNeeded函数，而第二个参数是保存应用进程的启动参数的argsForZygote。

zygoteSendArgsAndGetResult函数主要做的就是将传入的应用进程的启动参数argsForZygote，写入到ZygoteState中，ZygoteState其实是由openZygoteSocketIfNeeded函数返回的。

注释1处会调用ZygoteState的connect函数与名称为ZYGOTE_SOCKET的Socket建立连接，这里ZYGOTE_SOCKET的值为“zygote”，这是因为Zygote进程启动过程时，在Zygote的main函数中会创建name为“zygote”的Server端Socket。

注释2处如果连接name为“zygote”的Socket返回的primaryZygoteState与当前的abi不匹配，则会在注释3处连接name为“zygote_secondary”的Socket。这两个Socket区别就是：name为”zygote”的Socket是运行在64位Zygote进程中的，而name为“zygote_secondary”的Socket则运行在32位Zygote进程中。既然应用程序进程是通过Zygote进程fock产生的，当要连接Zygote中的Socket时，也需要保证位数的一致。

3、ZygoteInit

Socket进行连接成功并匹配abi后会返回ZygoteState类型对象，在分析zygoteSendArgsAndGetResult函数中讲过，会将应用进程的启动参数argsForZygote写入到ZygoteState中，而 ZygoteState 向 /dev/socket/zygote 写入参数，此时内部 LocalSocker 连接到 Zygote 持续监听，最终 Zygote 进程就会收到一个创建新的应用程序进程的请求。

Zygote 父进程会一直卡在 runSelectLoop 使用 epoll/select 等待 socket 事件，fork 出子进程时，会返回一个 Runnable caller。

开机优化：如果给 Zygote 添加参数 --enable-lazy-preload，Zygote 启动时会跳过预加载资源、预初始化 ICU、SQLite、WebView 等子系统步骤，只在第一次 fork 应用进程前再执行；第一次启动应用时会触发 preload，增加第一次 fork 的耗时。但会使得 首次启动应用变慢。

4、Zygote - Application

1、frameworks/base/core/java/com/android/internal/os/ZygoteServer.java

在 Zygote 主循环中使用 select() 等待 socket 事件，当 AMS 请求启动进程时，接受连接，创建一个 ZygoteConnection 对象，并调用 processOneCommand() 处理一次 fork 请求。

2、freworks/base/core/java/com/android/internal/os/ZygoteConnection.java#processOneCommand

从 Zygote socket 读取启动参数并解析成 Arguments，如果是 fork 请求，调用 Zygote.forkAndSpecialize() 创建子进程，在子进程中返回 handleChildProc()；在父进程中返回 handleParentProc()。接着 ZygoteInit.zygoteInit()（普通应用进程），接着

调用 RuntimeInit.commonInit() 初始化运行环境（日志、时区、异常处理等）。
调用 RuntimeInit.nativeZygoteInit()（native 层 binder 初始化等）。
调用 RuntimeInit.applicationInit() 进入应用入口。

3、frameworks/base/core/java/com/android/internal/os/RuntimeInit.java

通过反射加载 ActivityThread 类，用 new MethodAndArgsCaller(method, argv) 封装调用其 main(String[]) 方法交回 ZygoteInit.main() 执行 caller.run()，这会直接调用 ActivityThread.main(argv)

6、Stop 阶段

ActivityThread#handleStopActivity

handleStopActivity方法调用了performStopActivityInner来处理Activity的Stop状态，并且在Stop之后隐藏了Activity的DecorView的显示。

ActivityThread#performStopActivityInner

performPauseActivityIfNeeded中上一节已经有调用过，是为了确保Activity调用了onPause，如果已经处于pause状态将会直接返回。接着看 callActivityOnStop方法。

ActivityThread#callActivityOnStop

里有个关键信息，就是onSaveInstanceState方法的调用时机。

之前在分析Pause阶段时，我们知道在Android Honeycomb以前onSaveInstanceState会在onPause方法调用之前回调，而在callActivityOnStop中进一步对onSaveInstanceState的调用时机做出了解释，总结如下:

1️⃣ 在Android Honeycomb之前，onSaveInstanceState在onPause调用之前被调用
2️⃣ 在Android Honeycomb之后，Android P之前，onSaveInstanceState在onStop调用之前被调用
3️⃣ 在Android P之后，onSaveInstanceState在onStop调用之后被调用

Activity#performStop

此方法中同样会进行preStop、stop和postStop的事件分发，并且使用Instrumentation来回调onStop方法。

Activity#onStop

onStop方法中需要停止UI的刷新，停止运行中的动画，以及隐藏自动填充。

7、Destroy 阶段

Destroy阶段App除了要完成Activity本身的销毁逻辑外，还需要通知System Server来处理Activity栈相关的逻辑。

ActivityThread#handleDestroyActivity

此方法首先调用了 performDestroyActivity 执行了本地的销毁逻辑，然后通知 AMS 更新系统服务。

ActivityThread#performDestroyActivity

Instrumentation.callActivityOnDestroy 回调用 Activity 的 performDestroy 方法，与之前的生命周期类似，performDestroy方法主要是分发preDestroy、destroy和postDestroy事件，并且会调用onDestroy生命周期回调。

Activity#onDestroy

onDestroy方法中主要是保证Activity销毁时已经关闭了所有的由此Activity管理的Dialog、Cursor和SearchDialog，可以避免一些内存泄漏。
