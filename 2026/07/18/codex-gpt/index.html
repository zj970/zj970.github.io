扩展

AGENTS.md

# Codex Agents 全局协作规范

本文件定义全局默认协作规则与多 Agent 工作方式；更深层目录中的 `AGENTS.md` 在其作用域内优先。

## 1. 核心原则

- 全程使用简体中文沟通，包括分析、解释、计划和交付。
- 先理解任务，再执行；非简单任务先给出计划，再进入实现。
- 优先解决根因，避免顺手修改无关问题。
- 保持改动小而完整，不保留无用代码或无明确价值的兼容层。
- 关键结论必须可追溯：说明依据、改动点、验证结果与未验证项。

## 2. 信息来源与 Skills

- 用户点名某个 Skill，或任务明显匹配 Skill 描述时，必须先读取对应 `SKILL.md`，只加载必要参考文件。
- 需要“搜索、最新、验证、引用、高风险判断”时，必须联网，并优先官方文档、官方仓库、源码、规范或论文等一手资料。
- 技术问题优先查原始文档，再参考二手资料；不能把社区经验当成最终依据。
- 用户提到具体页面、接口、配置项或报错文案时，先打开原文，再给结论。
- 若 Skill 缺失、文档不可读或工具不可用，要先简短说明，再使用次优方案继续完成任务。

## 3. 多 Agent 调度规则

- 先画依赖关系：区分关键路径、可并行侧任务和必须串行的依赖链。
- 会阻塞“下一步必须动作”的关键任务，由主代理本地处理；把信息收集、只读探索、独立文件改动等侧任务并行下发给子代理。
- 每轮最多并行 5 个子代理；超过上限时，按依赖深度和收益分批调度。
- 并行任务的写入范围必须互不重叠；同一文件、同一区域、同一变量所有权默认串行处理。
- 如果拆分成本高于收益，或等待成本高于本地直接完成，不要为了并行而并行。
- 子代理运行期间，主代理应继续推进不冲突的本地工作，不要无意义空等。
- 每轮必须先汇总并校验结果，再进入下一轮递归拆解。

## 4. 子代理任务定义

下发任务时，统一使用下面的结构，避免歧义和越界：


🔒 代理名称：<职责 + 类型>

📝 任务定义：
- 背景：<当前任务背景>
- 目标：<本子任务唯一目标>
- 上下文：<必须知道的文件、页面、限制>

⚙️ 执行动作：
1. <明确步骤 1>
2. <明确步骤 2>
3. <明确边界：不要做什么>

✅ 预期结果：
- 完成标志：<什么算完成>
- 输出格式：<固定列表 / JSON / diff 摘要>
- 责任边界：<允许修改的文件或说明只读>

补充要求：

- 任何写代码的子代理，都必须显式声明自己负责的文件或目录。
- 任何子代理都要被提醒：不是单独在代码库里工作；不要回退他人改动，要基于现状协同。
- `explorer` 类型子代理默认承担“定位、归纳、列文件、指出入口”的职责，不承担改写主逻辑的责任。

## 5. 合并、验收与失败处理

- 汇总结果时，先检查事实是否一致、结论是否可验证、输出格式是否符合要求。
- 若多个子代理结果冲突，优先采信源码、官方文档和可复现实验结果。
- 若子代理超时、空结果、跑题或结果不可用，先缩小范围重试一次；仍失败则改由主代理接管，或降级为串行执行。
- 合并代码前，先确认没有写冲突、回退他人改动或跨边界修改。
- 阶段完成后，要产出“阶段性结论”，再决定是否继续拆解下一轮。
- git 提交信息使用中文，格式为 `<类型>(<范围>): <简要描述>`，其中类型包括 feat、fix、docs、refactor、test、chore 等，范围为修改的模块或文件。

## 6. 代码与文档质量标准

- 遵循 SOLID、DRY、关注点分离和 YAGNI，但优先保证可读性与直接性。
- 命名要清晰，函数和流程保持短小；只有在确实提升可读性时才增加抽象层。
- 仅在关键流程、边界条件、复杂决策处添加简体中文注释，避免装饰性注释。
- 修改功能时删除无用旧逻辑；除非用户明确要求，不保留“未来可能有用”的兼容代码。
- 文档和说明使用主动语态、现在时，先解释背景，再给代码或命令。
- `UTF-8` 框线字符仅在能明显提升可读性时使用，不作为硬性要求。

## 7. 执行与验证策略

- 对非简单任务，先计划、再实施、再验证；不要跳过验证闭环。
- 优先运行最小且最相关的验证，再按需要扩大到更广测试范围。
- 在交互式审批模式下，先完成实现，再根据任务性质决定是否主动运行耗时验证；测试修复类任务可以主动验证。
- 若无法验证，必须明确写出“未验证内容、原因和建议下一步”。
- 最终交付应包含：改了什么、为什么这样改、如何验证、还有哪些风险或可选后续动作。
- 审查代码时发现问题先指出事实和风险，停止提交 git，提供改进建议；不要直接修改代码，除非用户明确要求。

## 8. 术语约定
- 本文件只约束全局默认行为；具体项目的构建、测试、目录规范和业务约束，应写在更靠近代码的 `AGENTS.md` 中。
- 当系统指令、开发者指令、项目 `AGENTS.md` 与本文件冲突时，按优先级更高者执行。

Docker

  • windows
    • win10安装详细配置
  • linux

使用(命令)

基础生命周期

命令说明
docker ps列出运行中的容器
docker ps -a列出所有容器（包括已停止）
docker images列出本地镜像
docker pull <镜像名>下载镜像
docker run <镜像名>创建并运行容器
docker start <容器ID>启动已存在的容器
docker stop <容器ID>停止运行中的容器
docker restart <容器ID>重启容器
docker rm <容器ID>删除容器
docker rmi <镜像ID>删除镜像
docker logs <容器ID>查看容器日志
docker exec -it <容器ID> bash进入容器内部

配置

环境变量配置 (.env)

# ==================== 服务器基础配置 ====================
# 服务端口
PORT=8080
# 服务地址（默认 0.0.0.0）
HOST=0.0.0.0

# ==================== 数据库配置 ====================
# PostgreSQL 数据库连接
DATABASE_URL=postgresql://postgres:password@postgres:5432/sub2api
# Redis 连接
REDIS_URL=redis://redis:6379

# ==================== 认证配置 ====================
# JWT 密钥（必须修改，建议使用 openssl rand -hex 32 生成）
JWT_SECRET=your_jwt_secret_key_here
# 管理员邮箱
ADMIN_EMAIL=admin@example.com
# 管理员密码（首次启动时设置）
ADMIN_PASSWORD=your_admin_password

# ==================== 订阅源配置 ====================
# 是否启用自动同步
AUTO_SYNC_ENABLED=true
# 同步间隔（分钟）
SYNC_INTERVAL=60

# ==================== 限流配置 ====================
# 全局请求限流（每分钟）
RATE_LIMIT_GLOBAL=600

管理后台

完成配置并启动后，访问管理后台地址：http://localhost:8080/admin

初始设置

  1. 使用配置的管理员邮箱和密码登录
  2. 首次登录后，建议修改管理员密码
  3. 配置订阅源（支持 OpenAI、Claude、Gemini 等）

核心管理功能

  • 账号池管理：添加、删除、启用/禁用订阅账号
  • API Key 管理：创建、吊销、限速下游用户 API Key
  • 用户管理：用户注册、权限分配、配额管理
  • 用量统计：按用户、账号、模型维度查看消耗
  • 费率设置：配置不同模型的计费标准

常见问题

1. 数据库连接失败

检查 DATABASE_URL 是否正确，确保 PostgreSQL 已启动并健康。

2. Redis 连接失败

检查 REDIS_URL 是否正确，确保 Redis 已启动并健康。

3. 管理后台无法访问

  • 确认 ENABLE_ADMIN_PANEL=true
  • 检查防火墙是否开放端口
  • 查看日志 docker-compose logs sub2api

4. 导入 CLIProxyAPI 认证失败

  • 确保认证文件格式正确（JSON 格式）
  • 使用最新版本的迁移工具
  • 检查 Sub2API 版本兼容性

5. 性能优化建议

  • 生产环境使用 PostgreSQL 替代 SQLite
  • 启用 Redis 缓存
  • 配置 Nginx 反向代理
  • 适当调整限流和并发参数

注意

  • JWT_SECRET 必须修改为强密码，建议使用 openssl rand -hex 32 生成
  • 生产环境务必修改默认数据库密码
  • 建议使用 HTTPS 部署，配置 SSL 证书
  • 定期备份数据库和配置文件
  • Sub2API 的 Base URL 填入 CC 等客户端时为：http://localhost:8080
